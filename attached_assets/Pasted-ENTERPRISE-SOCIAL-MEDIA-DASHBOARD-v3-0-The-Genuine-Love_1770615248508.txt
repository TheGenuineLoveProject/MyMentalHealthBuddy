ENTERPRISE SOCIAL MEDIA DASHBOARD v3.0
(The Genuine Love Project — Replit Safe)

ABSOLUTE RULES:
- ADDITIVE ONLY. Do not delete, rename, or refactor existing files.
- No auto-posting to any social platform.
- No social platform API keys.
- Manual, human-in-the-loop only.
- Trauma-informed, educational tone enforced.
- If something already exists, EXTEND it — do not replace.
- All admin routes must be admin-auth protected.
- Every workflow must end with a PASS/FAIL checklist.

GOAL:
Upgrade the existing social media admin dashboard into an enterprise-grade
Narrative Operations Console that supports daily ethical publishing.

–––––––––––––––––
PHASE 0 — DISCOVERY
–––––––––––––––––
1. Locate existing social admin files:
   - client/src/pages/admin/social*
   - server/routes/social*
   - any social_posts storage (json or db)

2. Report what exists BEFORE modifying anything.

–––––––––––––––––
PHASE 1 — CANONICAL DATA MODEL (NO DUPLICATES)
–––––––––––––––––
Create or confirm ONE canonical source of truth:

Option A (Preferred): Database table
Option B: /content/narrative/social_posts.json (if DB not wired yet)

Each social post MUST include:

- id (uuid)
- title (internal)
- theme (enum or string)
- origin_type: blog | newsletter | reflection | campaign | standalone
- origin_id (nullable)
- captions:
    - instagram
    - tiktok
    - x
    - youtube
- hashtags (array)
- gentle_cta_url (nullable)
- safety_note (required)
- crisis_link_required (boolean)
- status:
    draft | review | approved | posted
- posted_platforms (array)
- created_by
- created_at
- reviewed_at
- approved_at
- posted_at

NO FIELD MAY BE OPTIONAL EXCEPT origin_id.

–––––––––––––––––
PHASE 2 — SERVER ROUTES (ADMIN ONLY)
–––––––––––––––––
Create or extend:

/api/admin/social/posts
- GET (filter by status, theme, origin)
/api/admin/social/post
- POST (create draft)
/api/admin/social/post/:id
- PUT (edit draft)
/api/admin/social/post/:id/submit
- POST → status = review
/api/admin/social/post/:id/approve
- POST → status = approved
/api/admin/social/post/:id/mark-posted
- POST → requires platform list, sets posted_at

ENFORCE:
- Cannot approve without safety_note
- Cannot mark posted unless approved
- Crisis link enforced when flagged

–––––––––––––––––
PHASE 3 — FRONTEND: ENTERPRISE DASHBOARD
–––––––––––––––––
Upgrade existing /admin/social page.

Required UI sections:

1. Pipeline Board
   - Columns: Draft → Review → Approved → Posted
   - Status transitions are button-based (no drag to skip)

2. Post Detail Panel
   - Shows narrative lineage
   - Shows safety note
   - Shows CTA destination
   - Shows timestamps

3. Platform Copy Panel
   - Tabs per platform
   - Character count helper
   - Copy-to-clipboard button
   - NO posting buttons

4. Signal Panel (Read-only)
   - Shows top themes from analytics_events
   - Shows recent high-performing blog links
   - “Suggested narrative focus” (rule-based, not AI)

5. Audit Panel
   - Shows who created, reviewed, approved, posted
   - Immutable log

–––––––––––––––––
PHASE 4 — SAFETY GUARDRAILS
–––––––––––––––––
Before approval:
- Validate no medical claims
- Validate no urgency language
- Validate no shame framing
- Validate CTA is gentle

If validation fails:
- Block approval
- Return human-readable error

–––––––––––––––––
PHASE 5 — VERIFICATION
–––––––––––––––––
PASS/FAIL checklist:

[ ] Admin can create draft
[ ] Draft cannot skip review
[ ] Approval requires safety note
[ ] Post cannot be marked posted without approval
[ ] Copy captions works
[ ] No auto-posting exists
[ ] No API keys present
[ ] All actions logged
[ ] Daily workflow < 20 minutes

STOP if any item fails and report.

FINAL OUTPUT REQUIRED:
- Files touched (paths)
- Routes added
- Data model used
- Screenshot list required
- Verification result
