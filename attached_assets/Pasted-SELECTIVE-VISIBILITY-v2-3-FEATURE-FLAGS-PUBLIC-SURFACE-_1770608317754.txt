SELECTIVE VISIBILITY v2.3 — FEATURE FLAGS + PUBLIC SURFACE GUARDBAND (REPLIT SAFE)

HARD RULES:
- ADDITIVE ONLY. Do not rename/delete/move/refactor existing files.
- Do NOT remove pages. We hide via flags, not deletion.
- Must be manual-first. No background autonomy.
- Must not change authentication or billing logic.
- Must end with VERIFIED via npm run verify.
- Must preserve Replit single-port rules.

GOAL:
1) Prevent unfinished pages/components from appearing in nav/CTAs or being promoted publicly.
2) Allow internal access (admin/dev) for work-in-progress pages.
3) Create a consistent user experience: only “Ready” modules visible publicly.
4) Provide a single canonical place to control visibility: a Feature Flag Registry.

PHASE 0 — BASELINE VERIFY (NO CHANGES)
Run:
- npm run verify
If FAIL, STOP and report.

PHASE 1 — CREATE CANONICAL FEATURE FLAG REGISTRY (SOURCE OF TRUTH)
Create:
- shared/featureFlags.mjs (or shared/featureFlags.ts if shared is TS-first; prefer .mjs if mixed)
Export:
- FEATURE_FLAGS object
- helper: isEnabled(flagName, userContext)

Structure example:
FEATURE_FLAGS = {
  public: {
    newsletter: { enabled: true, status: "ready" },
    blog: { enabled: true, status: "ready" },
    tools: { enabled: true, status: "ready" },
    community: { enabled: true, status: "beta" },
    healingJourneys: { enabled: false, status: "wip" },
    contentStudio: { enabled: false, status: "wip" },
    studyVault: { enabled: false, status: "wip" }
  },
  admin: {
    socialPublishing: { enabled: true, status: "ready" },
    billingDashboard: { enabled: true, status: "ready" }
  }
}

Rules:
- "ready" = visible publicly + nav + CTA
- "beta" = visible but labeled “Beta”
- "wip" = hidden from nav + CTA, but route may exist behind guards
- "disabled" = hidden and route blocks

PHASE 2 — CREATE CLIENT FEATURE FLAG CONTEXT (READ-ONLY FLAGS)
Create:
- client/src/contexts/FeatureFlagContext.jsx
Behavior:
- Loads default FEATURE_FLAGS from shared (static import)
- Optionally merges with server-provided flags if endpoint exists (do not require it)
Provide hook:
- useFeatureFlags()

PHASE 3 — ADD PUBLIC SURFACE GUARDS (NAV + CTA + FOOTER)
Goal: no “dead links” from the public surface.

Find primary navigation components:
- client/src/components/layout/TglpNavbar (or closest equivalent)
- Footer component(s)
- CommandCenter / ModulesPanel if present

Implement:
- Only render links when enabled
- If status === "beta", show “Beta” pill label
- If disabled/wip, hide entirely from public nav
- Add safe fallback: if a link exists but is wip, route to /coming-soon?feature=XYZ

Create:
- client/src/pages/ComingSoon.jsx
- Route: /coming-soon

ComingSoon page must:
- Explain “This feature is currently in development”
- Provide 3 safe next actions:
  - /tools
  - /newsletter
  - /blog
  - /crisis
- No guilt/urgency language

PHASE 4 — ROUTE-LEVEL GUARDS (OPTIONAL BUT RECOMMENDED)
Create:
- client/src/components/guards/FeatureGuard.jsx
Usage:
<FeatureGuard flag="healingJourneys" fallback="/coming-soon?feature=healingJourneys">
  <HealingJourneysPage />
</FeatureGuard>

Rules:
- public user sees ComingSoon
- admin user can bypass (if you already have admin context; otherwise skip admin bypass)

Do NOT break routing. Only wrap routes where needed.

PHASE 5 — VISIBILITY REGISTRY REPORT (PROOF + AUDITABILITY)
Create:
- docs/reports/SELECTIVE_VISIBILITY_STATUS.md
Include:
- Table of major modules (newsletter/blog/tools/community/healing/content studio/etc.)
- Flag name
- status
- where it is linked in nav/footer
- route path(s)
- public visible? yes/no
- admin visible? yes/no

PHASE 6 — VERIFICATION
Run:

Additionally:

- Confirm nav/footer no longer shows disabled items
- Confirm no broken CTA buttons remain for disabled features
