PUBLISHING v2.0 — SIGNAL-DRIVEN REFINEMENT (ETHICAL, MANUAL-FIRST)

HARD RULES:
- ADDITIVE ONLY. Do not rename/delete/move/refactor existing files.
- No background autonomy. No auto-posting. No manipulative growth mechanics.
- Must work on Replit: single port, deterministic startup.
- No PII logging. No storing raw IPs. No invasive tracking.
- Must end with PASS/FAIL evidence + VERIFIED via npm run verify.

GOAL:
1) Track a minimal set of ethical engagement signals (page views, CTA clicks, newsletter subscribes, upgrade intent)
2) Attribute signals to publishingRegistry items (blog/newsletter/social) via UTM + registry id
3) Produce weekly “What worked” report inside Admin (human decides changes)
4) Use signals to refine future editorial calendar (suggestions only, not auto changes)

PHASE 0 — BASELINE VERIFY (NO CHANGES)
Run:
- npm run verify
If FAIL, STOP.

PHASE 1 — DEFINE SIGNALS (MINIMAL, ETHICAL)
Signals to support:
- page_view (blog pages only, not mental health journaling private tools)
- cta_click (newsletter/pricing/blog/tools/crisis)
- newsletter_subscribe
- checkout_click (not purchase)
Rules:
- Do NOT track journal content, mood data, or chat content.
- Do NOT record message text.
- Do NOT store user agent string raw; store a short hash if needed.

PHASE 2 — UTM + REGISTRY ID STANDARD
Standard query params:
- utm_source
- utm_medium
- utm_campaign
- glp_id (publishingRegistry.id)

Update share links and CTA links generation to include glp_id when available.
Where:
- client/src/components/publishing/PublishingCtaBar.jsx (from v1.9)
- share components (ReflectionCard / etc.) only if they already generate URLs

PHASE 3 — DATA STORAGE (DB PREFERRED)
If analytics tables exist, extend; else add:
- publishing_signals:
  id, event_type, glp_id, path, cta_target, created_at,
  user_id (nullable), session_id (nullable), anon_id (nullable)
Rules:
- anon_id rotates daily (derived hash) so it cannot become a long-lived tracker.
- user_id only for logged-in admin analytics; never shown publicly.

PHASE 4 — COLLECTION ENDPOINTS
Create:
- server/routes/publishing-signals.mjs
Endpoints:
- POST /api/signals (public, rate limited)
- GET /api/admin/signals/summary (admin-only)
Validation:
- zod schema validate event_type, glp_id format, cta_target allowed list
Security:
- rateLimit required
- deny large payloads
- no arbitrary metadata

PHASE 5 — CLIENT HOOK (LIGHTWEIGHT)
Create:
- client/src/hooks/usePublishingSignals.ts
Features:
- fire page_view on blog page mount only
- fire cta_click on CTA button click
- debounce to avoid spam
- falls back gracefully if endpoint fails

PHASE 6 — WEEKLY INSIGHTS REPORT (ADMIN)
Create:
- server/services/publishingInsights.mjs (summarizer)
- client/src/pages/admin/AdminPublishingInsights.jsx
Route: /admin/publishing/insights
Shows:
- Top 10 posts by CTA clicks
- Top 10 by newsletter subscribes
- Pillar performance breakdown
- “Next week suggestions” (human review)
Rules:
- suggestions are text only; must NOT auto-edit calendar or registry

PHASE 7 — WEEKLY REPORT SCRIPT (OPTIONAL)
Create:
- scripts/generate-weekly-publishing-report.mjs
Output markdown to:
- docs/reports/publishing-weekly-YYYY-MM-DD.md
Include:
- metrics
- what worked
- what didn’t
- next experiments
No PII.
