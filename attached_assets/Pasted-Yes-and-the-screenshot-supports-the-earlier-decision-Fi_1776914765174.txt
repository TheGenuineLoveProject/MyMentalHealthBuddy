Ownership matrix

Here is the decision system for “one exact home.”

server/ai/*

Owner: AI/behavior engine
Purpose: reasoning, routing, memory, personalization, intervention selection, provider execution
Examples: orchestrator, provider, scoring, memory, moduleRouter, tools, aiTelemetry

server/routes/*

Owner: transport layer
Purpose: HTTP only
Rule: route files may validate, authorize, call orchestrator/service, and return JSON. They do not own business logic.

server/logging/*

Owner: observability layer
Purpose: platform/shared log sinks and structured logging
Rule: no product logic here

server/services/*

Owner: reusable product services
Purpose: non-AI domain behavior like reminders, content publishing, streak service, emails, monetization workflows

server/middleware/*

Owner: request guardrail layer
Purpose: auth, CSRF, rate limiting, request validation

server/utils/*

Owner: low-level utilities
Purpose: pure helpers and infrastructure helpers with no domain ownership

content/*

Owner: editorial/content system
Purpose: blogs, newsletters, guides, social scripts, SEO briefs, tool copy

client/features/*

Owner: product UX
Purpose: one feature one folder
Examples: chat, tools, onboarding, billing, streaks, insights

data/*

Owner: runtime state
Purpose: file-backed memory, summaries, profiles, logs
Rule: never mix source code here

What to do with the huge route list

Your screenshots show route sprawl. The fix is not “delete everything.”
The fix is route-family ownership.

Canonical route families

server/routes/
  ai.mjs                 # chat, tool execution, history, module entrypoints
  auth.mjs               # login, logout, session, oauth
  billing.mjs            # plans, checkout, portal, webhooks
  content.mjs            # blog feed, newsletter feed, guides
  analytics.mjs          # event ingest, summaries
  admin.mjs              # admin dashboard + privileged ops
  health.mjs             # /health /ready /version
  user.mjs               # profile, preferences, streaks, saved items

Everything else should eventually either:

* merge into one of those families, or
* move into services/ if it is not an HTTP boundary concern
