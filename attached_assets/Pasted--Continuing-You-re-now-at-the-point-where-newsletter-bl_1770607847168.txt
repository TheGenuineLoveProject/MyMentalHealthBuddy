PUBLISHING v1.9 — PILLAR ENGINE + EDITORIAL CALENDAR (CANONICAL, NO DUPES)

HARD RULES:
- ADDITIVE ONLY. Do not rename/delete/move/refactor existing files.
- No background autonomy. No cron. Manual-first.
- MUST integrate with duplicate prevention tooling (scripts/prevent-duplicates.mjs).
- All publishing artifacts must have ONE canonical source of truth.
- End with PASS/FAIL evidence + VERIFIED: PASS/FAIL via npm run verify.

GOAL:
1) Create a canonical publishing registry (one source of truth)
2) Establish 12 content pillars + governance rules
3) Add a 4-week editorial calendar generator (drafts only)
4) Add admin UI surfaces to select “Today’s Post” and track status
5) Ensure every blog/newsletter/social CTA links correctly and is consistent

PHASE 0 — BASELINE VERIFY (NO CHANGES)
Run:
- node scripts/prevent-duplicates.mjs
- npm run verify
If FAIL, STOP.

PHASE 1 — CANONICAL PUBLISHING REGISTRY (SOURCE OF TRUTH)
Create:
- content/publishing/publishingRegistry.json
Schema (must be enforced by a validator script):
[
  {
    "id": "glp-post-YYYYMMDD-xxx",
    "type": "blog" | "newsletter" | "social",
    "pillar": "PILLAR_KEY",
    "title": "...",
    "slug": "...",
    "status": "idea" | "draft" | "review" | "approved" | "published",
    "primaryCta": "/pricing|/newsletter|/blog|/crisis|/tools",
    "createdAt": "ISO",
    "publishAt": "ISO|null",
    "owner": "admin|system",
    "sourceFile": "relative path or null",
    "notes": ""
  }
]
Rules:
- id unique
- slug unique for blog items
- Only ONE registry file; no duplicates elsewhere

PHASE 2 — 12 PILLARS (LOCKED TAXONOMY)
Create:
- content/publishing/pillars.json
Include 12 pillars with keys + purpose + safe boundaries:
Examples:
- EMOTIONAL_LITERACY
- SELF_COMPASSION
- BOUNDARIES
- HEALING_PRACTICES
- VALUES_AND_MEANING
- NERVOUS_SYSTEM
- RELATIONSHIPS
- DAILY_RITUALS
- REFLECTION_PROMPTS
- COMMUNITY_AND_SUPPORT
- GROWTH_AND_MASTERY
- PLATFORM_ORIENTATION
Each pillar must include:
- description
- audience stage (beginner/intermediate/advanced)
- forbidden claims list (no medical promises)
- suggested CTA types

PHASE 3 — EDITORIAL CALENDAR GENERATOR (DRAFT ONLY)
Create:
- scripts/generate-editorial-calendar.mjs
Behavior:
- Generates a 4-week schedule (28 days)
- Rotates pillars evenly (no pillar repeats within 48 hours)
- Creates registry entries as type=blog with status=draft
- Creates 1 newsletter draft per week with type=newsletter status=draft
- Creates 1 social post suggestion per day type=social status=draft
Output:
EDITORIAL_CALENDAR: PASS
Print a compact table: date | type | pillar | title stub | status

PHASE 4 — REGISTRY VALIDATOR (NO BROKEN PUBLISHING)
Create:
- scripts/validate-publishing.mjs
Checks:
- JSON valid
- ids unique
- slugs unique for blog
- primaryCta routes exist in wouter routes/redirects (or are known static pages)
- status transitions valid (published cannot go back to draft)
- no unsafe CTA for crisis content (crisis items must link /crisis or /safety)
Output:
PUBLISHING_REGISTRY_VALIDATE: PASS/FAIL

PHASE 5 — ADMIN UI: “Publishing Command Center”
Create:
- client/src/pages/admin/AdminPublishing.jsx
Route: /admin/publishing
Features:
- Table of registry items with filter (type/status/pillar)
- Button: “Generate 4-week calendar” -> calls a backend/admin endpoint OR instructs to run script
- Field: “Today’s featured post” selector (stores chosen id in a small config file or DB setting)
- Buttons: mark status (draft->review->approved->published) with confirmations
- Copy buttons for CTA links (for social/manual use)

No auto publish. Manual only.

PHASE 6 — BLOG + NEWSLETTER LINK CONSISTENCY (SOFT CTA UNIFICATION)
Create:
- client/src/components/publishing/PublishingCtaBar.jsx
Props:
- ctaTarget
- copyVariant (short/long)
Rules:
- Gentle language
- No urgency/shame
- Must support: /newsletter, /pricing, /blog, /crisis, /tools
Integrate into:
- Blog post template page
- Newsletter landing page
Only ONE CTA bar per surface.

PHASE 7 — BACKEND OPTIONAL SUPPORT (IF NEEDED)
If your admin pages fetch via API:
Create:
- server/routes/admin-publishing.mjs
Endpoints:
- GET /api/admin/publishing/registry
- POST /api/admin/publishing/registry (append draft)
- PATCH /api/admin/publishing/registry/:id (status update)
Guard:
- adminAuth required

If you choose file-only:
- Admin UI can read via server static read endpoint (admin-only), but keep minimal.

PHASE 8 — VERIFY + EVIDENCE
Update:
- scripts/verify.mjs to include:
  node scripts/validate-publishing.mjs
  node scripts/generate-editorial-calendar.mjs --dry-run
Run required:
- node scripts/validate-publishing.mjs
- node scripts/generate-editorial-calendar.mjs --dry-run
- node scripts/prevent-duplicates.mjs
- npm run verify

D) Next 3 steps only
