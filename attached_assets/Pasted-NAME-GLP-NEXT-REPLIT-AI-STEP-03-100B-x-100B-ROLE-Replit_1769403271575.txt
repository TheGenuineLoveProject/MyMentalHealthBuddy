========================================================
STEP 0 — DRY-RUN: WHAT WE WILL IMPLEMENT
========================================================

GOAL:
Require authentication AND “admin role” for any /admin route and any admin API route.

ASSUMPTIONS (SAFE DEFAULTS):
- You already have some auth mechanism OR you can add a minimal one.
- You can store role as “user.role” (string) or “isAdmin” boolean.
- If auth is incomplete, we add a temporary “dev admin bypass” ONLY in development,
  disabled automatically in production.

SECURITY PRINCIPLES:
- Deny by default
- Server enforces access (frontend hiding is not security)
- Minimal data stored; no secrets in client

========================================================
STEP 1 — PATCH 1 (MAX 5 FILES) — DRY-RUN PLAN
========================================================

FILES (Patch 1):
1) server/middleware/requireAuth.ts          (NEW)
2) server/middleware/requireAdmin.ts         (NEW)
3) server/routes/adminGuard.ts               (NEW)
4) server/index.ts (or server/app.ts)        (EDIT: mount guard)
5) client/src/routes/AdminRoute.tsx          (NEW) OR existing router guard file

WHAT EACH FILE DOES:

1) server/middleware/requireAuth.ts
- If request has valid session/JWT/user: attach req.user and next()
- Else: return 401 with safe message
- DEV-ONLY bypass allowed when:
  NODE_ENV !== "production" AND env ALLOW_DEV_ADMIN === "true"
  (This prevents accidental prod exposure.)

2) server/middleware/requireAdmin.ts
- Requires req.user exists AND role/admin flag present
- Else: return 403

3) server/routes/adminGuard.ts
- Exports a router that:
  - applies requireAuth + requireAdmin to:
    - /admin/* (server-rendered routes if any)
    - /api/admin/* (admin APIs)
- Central “single source of truth” for admin protection

4) server/index.ts (or server/app.ts)
- Mount:
  app.use("/api/admin", adminGuardRouter)
  app.use("/admin", adminGuardRouter)
- Ensure this happens BEFORE admin routes are registered (important)

5) client/src/routes/AdminRoute.tsx (or similar)
- Frontend UX guard:
  - checks /api/me (or existing session endpoint)
  - if not admin: route to /sign-in (or show “Access denied”)
- NOTE: frontend guard is convenience only; server is authoritative.

VALIDATION:
- Visiting /admin/* while logged out -> 401
- Logged in non-admin -> 403
- Logged in admin -> OK
- In production build, DEV bypass is disabled even if env variable exists

COMMANDS:
- npm run build
- hit endpoints:
  - curl /api/admin/health -> 401/403/200 appropriately

ROLLBACK:
- Remove adminGuard router mount lines (only)
- Delete new middleware files
