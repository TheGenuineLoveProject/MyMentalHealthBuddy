# Database Query Optimization Guide

## Overview

The MyMentalHealthBuddy platform includes comprehensive query optimization helpers designed for production use. This guide shows how to use these tools to improve database performance.

## Available Optimization Helpers

### 1. Query Caching (`cachedQuery`)

**Purpose**: Cache frequently accessed data in-memory to reduce database load.

**When to Use**:
- Reference data that changes infrequently (crisis resources, templates)
- Data that's accessed multiple times per request
- User preferences and settings

**Example Usage**:

```typescript
import { cachedQuery, QUERY_CACHE_TTL, queryCache } from './lib/queryOptimization';

// In routes.ts - Cache crisis resources
app.get("/api/crisis-resources",
  asyncHandler(async (req, res) => {
    const resources = await cachedQuery(
      'crisis-resources:all',
      () => storage.getCrisisResources(),
      QUERY_CACHE_TTL.REFERENCE // 30 minutes
    );
    res.json(resources);
  })
);

// Cache user preferences
const userPrefs = await cachedQuery(
  `user:${userId}:preferences`,
  () => storage.getUserPreferences(userId),
  QUERY_CACHE_TTL.ACTIVITY // 5 minutes
);

// Invalidate cache when data changes
app.post("/api/crisis-resources",
  requireAdmin,
  asyncHandler(async (req, res) => {
    const resource = await storage.createCrisisResource(data);
    queryCache.invalidate('crisis-resources:'); // Clear all crisis resource caches
    res.status(201).json(resource);
  })
);
```

**Cache TTL Presets**:
- `QUERY_CACHE_TTL.STATIC` (3600000ms / 1 hour) - Static data that rarely changes
- `QUERY_CACHE_TTL.REFERENCE` (1800000ms / 30 minutes) - Reference data like crisis resources
- `QUERY_CACHE_TTL.USER_SETTINGS` (300000ms / 5 minutes) - User preferences and settings
- `QUERY_CACHE_TTL.ANALYTICS` (180000ms / 3 minutes) - Analytics and aggregated data
- `QUERY_CACHE_TTL.ACTIVITY` (60000ms / 1 minute) - Recent activity data
- `QUERY_CACHE_TTL.REALTIME` (5000ms / 5 seconds) - Real-time data (minimal caching)

**Important Notes**:
- ⚠️ This is single-instance in-memory caching
- For multi-instance Autoscale deployments, use Redis instead
- Always invalidate cache when underlying data changes
- Cache keys should be descriptive and include version/user info

### 2. Query Batching (`QueryBatcher`)

**Purpose**: Batch multiple individual queries into a single database request.

**When to Use**:
- Loading related data for multiple items (e.g., user details for a list of posts)
- Resolving foreign keys in bulk
- Fetching metadata for multiple resources

**Example Usage**:

```typescript
import { QueryBatcher } from './lib/queryOptimization';

// Create a batcher for user lookups
const userBatcher = new QueryBatcher<string, User>(
  async (userIds) => {
    const users = await storage.getUsersByIds(userIds);
    const userMap = new Map<string, User>();
    users.forEach(user => userMap.set(user.id, user));
    return userMap;
  },
  { maxBatchSize: 100, maxWaitMs: 50 }
);

// Usage in routes - these will be batched automatically
app.get("/api/social-posts",
  requireAuth,
  asyncHandler(async (req, res) => {
    const posts = await storage.getSocialPostsByUserId(req.user!.id);
    
    // Batch load all unique authors
    const authors = await Promise.all(
      posts.map(post => userBatcher.load(post.authorId))
    );
    
    const postsWithAuthors = posts.map((post, i) => ({
      ...post,
      author: authors[i]
    }));
    
    res.json(postsWithAuthors);
  })
);
```

**Benefits**:
- Reduces N+1 query problems
- Decreases database roundtrips
- Automatically handles deduplication
- Configurable batch size and wait time

### 3. Paginated Queries (`paginatedQuery`)

**Purpose**: Efficiently handle paginated results with proper metadata.

**When to Use**:
- Listing resources (journals, mood entries, posts)
- Search results
- Activity feeds
- Any endpoint returning large result sets

**Example Usage**:

```typescript
import { paginatedQuery } from './lib/queryOptimization';

app.get("/api/journals",
  requireAuth,
  asyncHandler(async (req, res) => {
    const page = parseInt(req.query.page as string) || 1;
    const limit = parseInt(req.query.limit as string) || 20;
    
    // Note: You'll need to add offset/limit support to storage methods
    // or implement pagination in-memory for existing methods
    const allJournals = await storage.getJournalsByUserId(req.user!.id);
    
    const result = await paginatedQuery({
      page,
      limit,
      queryFn: async (offset, limit) => allJournals.slice(offset, offset + limit),
      countFn: async () => allJournals.length
    });
    
    res.json({
      journals: result.items,
      pagination: result.pagination
      // pagination includes: page, limit, total, totalPages, hasNext, hasPrev
    });
  })
);

// Better: Extend storage interface to support offset/limit natively
// interface IStorage {
//   getJournalsByUserId(userId: string, offset?: number, limit?: number): Promise<SelectJournal[]>;
//   getJournalsCountByUserId(userId: string): Promise<number>;
// }
```

**Features**:
- Automatic input validation (min page: 1, max limit: 100)
- Parallel execution of data query and count query
- Complete pagination metadata
- Built-in offset/limit calculation

### 4. Optimized Query Wrapper (`optimizedQuery`)

**Purpose**: Wrap queries with metrics logging and error handling.

**When to Use**:
- Monitoring query performance in production
- Identifying slow queries
- Debugging database issues
- Any critical query

**Example Usage**:

```typescript
import { optimizedQuery } from './lib/queryOptimization';

app.get("/api/mood-entries/:userId",
  requireAuth,
  asyncHandler(async (req, res) => {
    const entries = await optimizedQuery(
      'get-user-moods',
      async () => storage.getMoodEntriesByUserId(req.params.userId)
    );
    
    res.json(entries);
  })
);
```

**What It Logs**:
- ✅ Query duration for all queries
- ⚠️ **Slow query warnings** for queries taking >500ms
- ❌ Query errors with timing information
- 📊 Record counts for array results

**Example Output**:
```
[SLOW QUERY] get-user-moods took 785ms (1250 records)
[QUERY ERROR] fetch-social-profiles failed after 1200ms: Connection timeout
```

## Connection Pool Configuration

The database connection pool is already optimized for Replit Autoscale deployments:

```typescript
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 10,                      // Conservative max per instance
  min: 0,                       // CRITICAL for scale-to-zero
  idleTimeoutMillis: 30000,     // Close idle connections after 30s
  connectionTimeoutMillis: 5000, // Timeout acquiring connections
  allowExitOnIdle: true,        // Allow pool to exit when idle
  statement_timeout: 30000,     // Prevent runaway queries
  query_timeout: 30000          // Overall query timeout
});
```

**Why These Settings?**:
- `min: 0` - Essential for Autoscale scale-to-zero behavior
- `max: 10` - Prevents overwhelming the database with too many connections
- `allowExitOnIdle: true` - Allows instances to shut down when not in use
- `idleTimeoutMillis: 30000` - Aggressively closes unused connections

## Implementation Checklist

### For Existing Endpoints

When optimizing an existing endpoint:

- [ ] Identify frequently accessed data → Add `cachedQuery`
- [ ] Check for N+1 problems → Add `QueryBatcher`
- [ ] Large result sets → Add `paginatedQuery`
- [ ] Critical queries → Wrap with `optimizedQuery`
- [ ] Cache invalidation → Add `queryCache.invalidate()` to mutations

### For New Endpoints

When creating a new endpoint:

- [ ] Use `paginatedQuery` for list endpoints
- [ ] Use `cachedQuery` for reference data
- [ ] Use `optimizedQuery` for all database calls
- [ ] Plan cache invalidation strategy
- [ ] Consider batching for related data

## Migration to Redis (Future)

When deploying to multi-instance Autoscale:

1. **Install Redis client**:
   ```bash
   npm install ioredis
   ```

2. **Update QueryCache** to use Redis:
   ```typescript
   import Redis from 'ioredis';
   
   const redis = new Redis(process.env.REDIS_URL);
   
   class RedisQueryCache {
     async get<T>(key: string): Promise<T | null> {
       const cached = await redis.get(key);
       return cached ? JSON.parse(cached) : null;
     }
     
     async set<T>(key: string, data: T, ttl: number): Promise<void> {
       await redis.setex(key, Math.floor(ttl / 1000), JSON.stringify(data));
     }
     
     async invalidate(keyPrefix: string): Promise<void> {
       const keys = await redis.keys(`${keyPrefix}*`);
       if (keys.length > 0) {
         await redis.del(...keys);
       }
     }
   }
   ```

3. **Update cache calls** to use async/await:
   ```typescript

     'crisis-resources:all',
     () => storage.getCrisisResources(),
     QUERY_CACHE_TTL.REFERENCE
   );
   ```

## Performance Best Practices

### DO ✅

- Cache reference data with long TTL
- Batch related queries together
- Use pagination for large datasets
- Monitor slow queries
- Invalidate cache on mutations
- Set appropriate query timeouts

### DON'T ❌

- Don't cache user-specific data without user ID in key
- Don't cache frequently changing data
- Don't forget to invalidate caches
- Don't use in-memory cache in multi-instance deployments
- Don't skip pagination on large tables
- Don't ignore slow query warnings

## Common Patterns

### Pattern 1: Cached Reference Data

```typescript
// Good for: Crisis resources, templates, categories
const resources = await cachedQuery(
  'crisis-resources:all',
  () => storage.getCrisisResources(),
  QUERY_CACHE_TTL.REFERENCE
);
```

### Pattern 2: Paginated User Data

```typescript
// Good for: Journals, posts, activity logs
const allJournals = await storage.getJournalsByUserId(userId);
const { items, pagination } = await paginatedQuery({
  page: req.query.page || 1,
  limit: 20,
  queryFn: async (offset, limit) => allJournals.slice(offset, offset + limit),
  countFn: async () => allJournals.length
});
```

### Pattern 3: Batched Related Data

```typescript
// Good for: Loading authors for posts, categories for items
const userBatcher = new QueryBatcher(...);
const authors = await Promise.all(posts.map(p => userBatcher.load(p.authorId)));
```

### Pattern 4: Monitored Critical Queries

```typescript
// Good for: Analytics, reports, complex queries
const analytics = await optimizedQuery(
  'generate-user-analytics',
  async () => storage.generateUserAnalytics(userId)
);
```
