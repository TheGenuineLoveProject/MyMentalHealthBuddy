# b10 — Operational SOPs

## Purpose
Document a single Standard Operating Procedure for a recurring platform task — so anyone on the team can run it without guesswork.

## Output Contract
1. **SOP Title + Owner** (1 line each).
2. **Trigger** (1 line): when this SOP runs (event, schedule, threshold).
3. **Inputs** (bullet list): what's needed before starting.
4. **Procedure** (numbered steps): each step is one verb + one specific action.
5. **Tools / Links** (table): every tool, dashboard, or doc referenced.
6. **Outputs / Artifacts** (bullet list): what this SOP produces.
7. **Success Criteria** (3 bullets): measurable signs the SOP ran correctly.
8. **Failure Modes + Recovery** (2–3 bullets): what goes wrong, how to recover safely.
9. **Review Cadence** (1 line): when to re-read and update this SOP.

## Hard Rules
- Every step must be reproducible by someone new to the task.
- No tribal knowledge ("ask Sarah") — name the system, not the person.
- Crisis-related SOPs require dual review before publishing.
