# MyMentalHealthBuddy by The Genuine Love Project

## Overview
MyMentalHealthBuddy (MMHB) by The Genuine Love Project is an AI-powered mental wellness platform dedicated to fostering self-love, healing, and emotional growth. It offers AI-assisted guidance, mood tracking, journaling, crisis support, and evidence-based tools. The project's core mission is to provide an ethical, accessible, and personalized mental wellness solution by integrating AI with trauma-informed psychological principles to build emotional resilience.

### Technical Implementations
The project is a monorepo with a React 18 (TypeScript, Vite, Wouter, React Hook Form, Zod, Tailwind CSS, Lucide React) client and a Node.js/Express (TypeScript) server. The backend provides a RESTful API with middleware for security and session management. A trauma-informed NLP layer and "Wellness Microcopy Library" ensure supportive user interactions, integrating Social Work-Informed Frameworks. A Dual-Engine AI Prompt Architecture separates `healing/` (user-facing) and `business/` (admin/staff) engines with strict data boundaries, each having a `registry.json` and modular prompts. Core orchestration handles RBAC, crisis-first intent routing, PII redaction, and structured event logging. Live HTTP routes manage access to these engines. A Self-Tuning Fallback Library provides keyword-driven replies for offline or error scenarios, recording metrics for insights. Guest memory is handled ephemerally with `x-guest-id` headers for unauthenticated users. A self-evolving prompt registry lets admins hot-edit prompt files at runtime via `GET/PUT /api/ai/{healing|business}/admin/prompts/:id` (healing gated `requireAdult` â†’ `requireAuth` â†’ `requireAdmin`; business gated `authGuard` â†’ `requireAdmin`); writes are atomic, capped at 50 KB UTF-8, and append a JSONL audit entry (`ts, actor, engine, promptId, prevSha, newSha, bytes, reason`) to `ai/{engine}/_audit.log.jsonl`, viewable via `GET /api/ai/{engine}/admin/audit`. Engine whitelist (`healing|business`) enforced at `promptEngine.mjs`. A per-user structured profile layer (`server/ai/profileExtractor.mjs` + `server/ai/profileStore.mjs`) extracts six categories (`emotional_patterns, triggers, relationship_themes, coping_strategies, support_needs, risk_flags`) from chat history every 2 turns once history â‰Ą4 messages, merges (union/dedupe, cap 20 per category) into `data/memory-profile/<user>.json`, and the orchestrator injects a bullet-rendered version (char-capped at 4000) as a system message alongside the conversation summary; extraction is logged to telemetry as `route="/ai/internal/profile"` and `extra.purpose="profile_extraction"`. A per-user in-process async mutex (`server/utils/asyncLock.mjs`) wraps `saveMemory` to serialize the read-modify-write of memory + profile across the awaited summarizer/extractor calls, preventing concurrent-request races. A subtle-emotion inference layer (`server/ai/emotionInference.mjs`) is a pure function that scans the user's message for indirect/suppressed phrasing across 5 states (`anxiety, overwhelm, sadness, avoidance, self_doubt`) â€” complementary to `moduleRouter` (which selects intervention modules); inferred states feed only the policy's TONE branch. An adaptive response policy layer (`server/ai/responsePolicy.mjs`) is a pure function that converts `{risk, profile, scoring, modules, inferredStates}` into a style/intervention contract (`tone`, `verbosity`, `structure`, `interventions[]`, `constraints[]`); profile-driven tweaks include reassurance/grounding from `support_needs`, breathing+5-4-3-2-1 from anxiety patterns, sleep-hygiene tip from insomnia patterns, and "monitor for escalation cues" from any `risk_flags`; `risk.level === "high"` overrides interventions to grounding/safety/resource_signpost. The rendered policy is injected as the LAST system message in the provider (after summary + profile, before history) so it wins instruction precedence; telemetry records `policy: { level, tier, interventions: count, modules }`. An intervention module router (`server/ai/moduleRouter.mjs`) sits between the profile layer and the policy layer: `selectModules({profile, input})` chooses from 8 evidence-backed modules (anxiety, thought_pattern, emotional_processing, loop_detection, relationship, self_regulation, awareness, growth) using input regex (anxiety/thought-loop/overwhelm/relationship/values, with "can't" / "cant" / "cannot" / "can not" all matched) plus profile signals (`behavior_loops`, anxiety/insomnia patterns, `values`). Each `MODULE_REGISTRY` entry carries `interventions[]`, `tag`, `priority` (1â€“3), and an optional `structure`; `selectModules` dedupes candidates, sorts by priority desc, and caps at `MAX_MODULES=2` to prevent intervention soup. Selected modules resolve via `resolveModules(ids)` into additional interventions that layer on top of profile-driven tweaks; the first non-null `structure` in the resolved set wins (priority order) when risk is not "high", letting CBT-shaped, feeling-shaped, or loop-shaped flows replace the default structure. The provider injects an "Active support modules: â€¦" system message immediately before the policy message so the model sees which frameworks the policy is operationalizing. A tool execution layer (`server/ai/tools.mjs` catalog of 7 evidence-informed micro-exercises â€” `box_breathing, grounding_54321, thought_reframe, emotional_checkin, relationship_repair, pattern_interrupt, overload_reset` â€” each bound to a module id, with `+ server/ai/toolSelector.mjs` and `server/ai/toolExecutor.mjs`) sits after module selection: `selectTool({modules, profile, input})` picks at most one tool by walking the priority-sorted module ids and returning the first bound match (with profile-anxiety and input-overwhelm fallbacks for null-module cases); `executeTool(tool)` returns a structured payload `{ tool: {id,title,type,durationMin}, exercise: {intro,steps,closing} }` which the orchestrator (a) surfaces in `response.tool` for UI exercise cards on both AI-success and fallback paths and (b) passes to the provider, which injects it as an "Active exercise:" system message between profile and the framework hint so the model walks the user through the steps verbatim instead of paraphrasing. Telemetry adds `toolId`, `toolType`, and `moduleCount` per call. Crisis paths short-circuit before any tool selection so no exercise is ever injected on a safety route. Profile schema extended with three module-system fields (`core_beliefs`, `behavior_loops`, `values`) â€” `EMPTY_PROFILE` spread keeps old files forward-compatible. Profile rendering for prompt injection uses a priority order so safety-critical fields (`risk_flags`, `support_needs`) render first and survive the 4000-char cap if truncation fires; lowest-priority fields (`values`) drop first. A unified `server/engine/crisisDetection.mjs` facade exposes a single `detect(text, opts)` entry that wraps the existing safety stack (`detectCrisis` keywords, `assessRisk` levels, `safetyGuardInput` blocked-pattern guard, optional `classifyCrisis` LLM verdict) and returns `{ crisis, blocked, level, signals, response, escalate988, cleanText, source }`. `/api/ai/chat` runs the facade first on every message; crisis hits short-circuit with the canonical 988/741741 response and `escalate_immediately` action. Hardened security middleware protects routes with helmet, CORS, rate limiting, input sanitization, and security headers.

### Launch Surface (`/start`) & Soft Paywall
A dedicated launch surface at `/start` (`client/src/pages/Start.tsx`) gives new users a single-tap entry to one of three relief flows ("Calm Me Down", "Help Me Think Clearly", "Understand This Feeling"), each calling `/api/ai/chat` with a curated message. Successful, non-crisis responses surface a streak chip via `POST /api/streaks/checkin` (guest-safe: returns `{authenticated:false}` without erroring) and may surface a soft upgrade card. The soft paywall is gated server-side by `server/ai/paywallPolicy.mjs` (pure function â€” `shouldShowPaywall({isCrisis, successfulSessions, dailySessions, requestedPremium})` with thresholds at 3, plus `getPaywallReason` returning `value_proven | daily_limit | premium_feature | null`) and a file-backed counter `server/utils/usageCounter.mjs` (`data/usage/<sanitizedId>.json`, daily rollover via UTC date stamp, identity = `req.dbUserId || x-guest-id || "anonymous"`, sanitized to alphanumeric+`_-` and capped at 64 chars). Per-identity writes are serialized through `withLock("usage:<id>", ...)` from `server/utils/asyncLock.mjs` to prevent lost-update races on concurrent requests. The route layer (`server/routes/ai.mjs`) attaches `result.response.paywall = {show, reason}` only when `result.ok && !response.isCrisis && response.reply`, and the entire block is wrapped in try/catch so paywall accounting failure never breaks an AI response. Crisis paths return `paywall: null` (never attached). Frontend `PaywallCard` renders three reason-keyed copy variants and fires whitelisted telemetry events `paywall_shown` (on mount) and `paywall_clicked` (on Unlock Pro link to `/pricing` or "Keep using free tools" dismiss). A lightweight client-side telemetry funnel (`server/routes/telemetry.mjs` â†’ `POST /api/telemetry/event` â†’ `logEvent` in `server/ai/aiTelemetry.mjs` appending to `logs/events.jsonl`) is gated to a strict 6-event whitelist (`start_page_click`, `first_tool_selected`, `first_response_success`, `streak_incremented`, `paywall_shown`, `paywall_clicked`); unknown types are silently dropped, no message bodies are ever sent or stored, and all telemetry is fail-silent at both client and server so an outage cannot break the UI.

### Feature Specifications
Core features include AI-powered Chat Therapy and Wellness Tools (State Tracker, Journal Prompts). The platform offers specialized APIs for various intelligence domains (e.g., Knowledge Synthesis, Trauma Healing Protocols, Spiritual Intelligence). The brand color palette ("Enterprise Elite II") uses specific gold-sage gradient lines for section dividers. Monetization is via a four-tier subscription model using Stripe. Security features include rate limiting, CSP, and input sanitization. Engagement tools comprise gamification and a Content Studio. User features include daily healing reminders, voice affirmations, and an AI companion. Admin tools provide dashboards for health, social studio, and command center. Content is organized via a Learning Hub and a unified `blog_posts` table for all publishable content with CRUD operations and RSS feeds.

### System Design Choices
Drizzle ORM is used with a Neon PostgreSQL database, employing UUIDs and indexed foreign key constraints. Production security includes CORS allowlisting, JWT authentication, Helmet, and rate limiting. The system has an observability layer with health, system, and kernel endpoints for telemetry and integrity checks. A `Prompt-OS Execution Prompt Library` ensures canonical prompt modules, validated against `promptspec.schema.json`. Production readiness features include a 503 readiness gate, health probes, telemetry parity, request tracing, and hardened administration access, governed by a `CHANGE_GATE` protocol and `Component Registry`.
